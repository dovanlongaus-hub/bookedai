# BookedAI Sprint 1 Owner Execution Checklist

Date: `2026-04-17`

Document status: `active sprint checklist`

## 1. Purpose

This document is the real execution checklist for Sprint 1, organized by owner.

It exists to convert the Phase 0 plan into practical execution work.

Sprint 1 is not considered complete only because documents exist.

Sprint 1 is complete only when each owner finishes their checklist and the cross-functional outputs align.

## 2. Sprint 1 mission

Finish the Phase 0 reset and lock the project onto one shared execution model for the upgraded BookedAI product.

## 3. Sprint 1 success criteria

- Phase 0 core documents are aligned
- BookedAI is consistently framed as a multi-channel AI revenue engine
- pricing is consistently framed as setup fee plus performance-based commission
- the standard execution formula is documented
- Phase 0 has a completed Epic -> Story -> Task backlog
- Sprint 1 has a real owner checklist and closure model

## 4. Owner checklist

## Owner: Product lead

### Required outcomes

- product definition approved
- narrative and pricing language approved
- phase and sprint framing approved

### Checklist

- review and approve PRD core positioning
- review and approve headline, one-liner, and commercial message hierarchy
- review and approve target audience and pains
- review and approve pricing positioning
- review and approve mandatory landing-page sections
- review and approve Phase 0 acceptance criteria
- review and approve handoff into Phase 1

### Completion evidence

- approved comments or signoff on PRD and Phase 0 plan

## Owner: Solution architect

### Required outcomes

- architecture reflects the upgraded product model
- master execution model is coherent

### Checklist

- review and approve target platform architecture reset
- review and approve solution architecture master execution plan
- confirm domain boundaries for revenue, attribution, missed revenue, recovery, and commission
- confirm product-surface responsibilities
- confirm phase dependencies and release gates
- confirm Phase 0 execution package completeness

### Completion evidence

- approved comments or signoff on architecture and master plan docs

## Owner: Product operations or PM

### Required outcomes

- roadmap and sprint sequencing are executable

### Checklist

- review implementation roadmap
- review MVP sprint execution plan
- verify sprint ordering matches the new product direction
- update sprint document register links and sequencing
- verify Phase 0 backlog is actionable
- verify Sprint 1 deliverables are explicit

### Completion evidence

- approved comments or signoff on roadmap, sprint plan, and sprint register

## Owner: Frontend lead

### Required outcomes

- public rebuild is implementation-ready

### Checklist

- review public brand and landing-page blueprint
- review `docs/architecture/landing-page-system-requirements.md`
- review design tokens and UI block definitions
- review component tree and file mapping assumptions
- verify the proposed structure fits the current frontend repo without a rewrite-first move
- flag any risky implementation assumptions early
- approve Phase 1-2 package from a frontend feasibility standpoint

### Completion evidence

- approved comments or signoff on Phase 1-2 package and public implementation feasibility

## Owner: Backend lead

### Required outcomes

- pricing and public product narrative do not get ahead of backend reality

### Checklist

- review pricing strategy implications on data and APIs
- review required commercial vocabulary
- confirm which later widget concepts are safe as public previews
- flag unsupported reporting claims
- confirm attribution and commission implications are reflected in later phases
- approve the Phase 1-2 instrumentation assumptions at a high level

### Completion evidence

- approved comments or signoff on pricing and commercial-domain implications

## Owner: QA or release owner

### Required outcomes

- governance and release-truth rules are in place

### Checklist

- review change governance document
- confirm narrative truth gate is documented
- confirm commercial truth gate is documented
- confirm Sprint 1 closure criteria are explicit
- confirm future release discipline expects documentation write-back
- approve the standard formula for phase execution and sprint owner checklists

### Completion evidence

- approved comments or signoff on governance and release-truth rules

## 5. Cross-functional review checklist

- PRD, architecture, roadmap, sprint plan, pricing, and public blueprint do not conflict
- Phase 0 backlog matches Phase 0 execution plan
- Sprint 1 owner checklist matches actual owner responsibilities
- standard execution formula is written into governance docs
- next-phase implementation can start without strategic ambiguity

## 6. Sprint 1 closeout checklist

- mark Sprint 1 documents as current and aligned
- record any unresolved assumptions
- create explicit Phase 1 implementation kickoff owner list
- create Sprint 2 start criteria
- confirm no conflicting narrative remains in core planning docs

## 7. Sprint 2 start criteria

Sprint 2 can start only when:

- Product approves positioning and pricing baseline
- Architecture approves target domain and phase model
- Frontend approves implementation feasibility for the landing rebuild
- Backend confirms no unsupported public claims are being embedded as truth
- QA or release owner approves governance and truth-gate baseline
