# 2026-05-01

- Merged `BookedAI CTO Master Spec v2` into `project.md` as the latest top-level requirement baseline.

- Re-reviewed the full requirement direction before BL-1 and added `docs/development/revenue-first-enterprise-requirements-refinement-2026-05-01.md`, plus stronger Google-like search, enterprise UX, lifecycle-complete, and revenue-first requirement language into `project.md` and the master PRD.
- Added a new scale/platform requirement layer: BookedAI now needs clear versioned API families, easy-install partner widgets, tenant-isolated AI ingestion from web/text/files into tenant-specific databases, independent tenant API/webhook behavior, and shared-portal-by-default with enterprise dedicated portal as an exception path.
- Added `docs/architecture/api-tenant-platform-architecture-spec-2026-05-01.md` and locked a business-model-first operating rule: BookedAI should prioritize simple but professional management for setup fees, subscriptions, booking-linked commission, and BookedAI revenue visibility before expanding into heavier finance or platform complexity.
- Added a stronger monetization rule to the pre-coding baseline: feature priority should follow setup-fee onboarding, subscription continuity, commission capture, tenant revenue visibility, and BookedAI revenue tracking before broader platform complexity.
- Locked the default go-forward posture as `run, collect money, manage booked revenue and BookedAI revenue clearly first`, with shared portal as default, dedicated tenant portal only as an enterprise exception, and API/widget/installability work treated as first-class roadmap requirements.
- Tightened the delivery order again: baseline sequencing now explicitly moves from simple to complex, with business control, cashflow visibility, setup/subscription collection, booking-linked commission truth, and operator revenue management placed ahead of broader platform depth, richer customization, and heavier enterprise complexity.
- Reordered the managed backlog so `simple commercial control and cashflow truth` comes before repeatable scale/installability foundations, and both come before deeper vertical or expansion work.
- Added `docs/architecture/business-model-and-partner-pricing-baseline-2026-05-01.md` and upgraded the monetization baseline from `setup fee + commission` to an explicit `setup fee + subscription + commission` model, with structured partner pricing posture and tighter sync between commercial model, roadmap, and functional system planning.
- Added `docs/development/bl-14a-monetization-and-commission-operating-spec-2026-05-01.md` and pushed BL-14A into explicit phase priority: it is now the first concrete commercial closure contract inside Phase 21 for tenant commercial profile, setup fee, subscription, commission truth, and operator commercial action queues.
- Added `docs/development/bl-14a-sub-slice-breakdown-2026-05-01.md` so BL-14A now has a concrete execution order inside Phase 21: commercial profile, setup/subscription state machine, commission pipeline, revenue summary/action queue, then audit and acceptance gate.
- Added an early infrastructure planning rule: frontend runtime consolidation, Postgres/Supabase scale discipline, `n8n` role boundaries, and Docker/container-first packaging must be decided in early project phases instead of being deferred until scale pain appears.
- Added `docs/architecture/early-technical-foundation-and-scale-architecture-plan-2026-05-01.md` and recommended the best current direction as `Next.js` long-term frontend target, `FastAPI` modular monolith backend, `Postgres` truth with self-hosted Supabase acceptable for now, `n8n` as glue only, and container-first execution. The whole project order is now explicitly re-read from technical foundation first, then container/env normalization, backend scale discipline, monetization layer, frontend consolidation, partner/installability, and only then broader scale breadth.
- Added `docs/architecture/bookedai-whole-system-rearchitecture-baseline-2026-05-01.md` and reset the whole-system execution order around the shortest money-making customer loop: searchable first, installable second, bookable third, full-loop operable fourth, monetization visible fifth, repeatable scale sixth, and deeper vertical/enterprise complexity later.
- Added `docs/architecture/phase-a-technical-foundation-decision-package-2026-05-01.md` to formalize the first rearchitecture gate: approve `Next.js` as the long-term frontend target, `FastAPI` modular monolith, `Postgres` truth with self-hosted Supabase operational packaging, `n8n` glue-only boundaries, and Docker/container-first service packaging.
- Added `docs/architecture/change-request-governance-baseline-2026-05-01.md` so BookedAI can absorb many feature requests, business ideas, upgrades, and scope changes without derailing the revenue-first baseline; meaningful changes now require explicit CR classification, baseline-alignment review, and a merge/defer/reject decision.
